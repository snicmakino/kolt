**Bugfix**

- `kolt add` no longer mutates `kolt.toml` when the fetch fails. A typo or unfetchable coordinate (404, dead repo) used to leave the project locked into a bogus entry until the user ran `kolt remove`; the new resolve-then-write order keeps `kolt.toml` byte-identical on failure (#353).
- `kolt add <ga>` (no version) no longer auto-picks pre-releases. The pre-fix code trusted Maven Central's `<release>` tag, which surfaces `-rc`, `-alpha`, `-beta`, `-M\d+`, `-eap`, `-dev`, `-preview` qualifiers — contradicting the documented "latest stable" contract. Resolution now parses the full `<versions>` list, filters by qualifier, and picks the highest stable. If every published version is pre-release, kolt falls back with a `warning: no stable release found ...` to stderr so brand-new artefacts aren't blocked (#354).
- Failed downloads now name the URL and HTTP status (or network-error message) for every repository tried, instead of collapsing to a bare `failed to download <ga>`. Distinguishing typo / 401 / 5xx / DNS no longer requires re-running under strace. Empty `[repositories]` surfaces a config-fix hint rather than a network-style message (#355).
- `kolt remove` of the last entry in a section now drops the section header. A bare `[dependencies]` no longer dangles in the diff — the function still only normalizes sections it actually removed from, so a pre-existing empty header survives untouched (#360).
- Unknown subcommand now exits `127` (`Command not found`) instead of `1` (`Build error`), matching the README exit-code table. CI scripts that grep for `$? == 1` to gate on a typo'd `kolt foo` should switch to `127` (#361).
- `kolt fmt` no longer leaks the 4-line `sun.misc.Unsafe::objectFieldOffset` deprecation warning ktfmt's bundled intellij-util triggers on JDK 23+. The fix passes `--sun-misc-unsafe-memory-access=allow` to the ktfmt JVM, gated to JDK 23+ where the warning fires (the flag is unrecognised on older JDKs, which never saw the warning either) (#358).
- `kolt build` / `kolt test` on a Native target (e.g. `target = "linuxX64"`) no longer leak `sun.misc.Unsafe` and `java.lang.System::load` warning blocks from konanc's bundled `kotlin-native-compiler-embeddable.jar` on JDK 23+. The same warnings fired once per konanc invocation across every stage that spawns the konanc JVM — each `[[cinterop]]` entry, the klib stage, and the kexe link stage — so a project with two cinterop blocks emitted 32 lines of noise per fresh `kolt build`. The fix passes `--sun-misc-unsafe-memory-access=allow` and `--enable-native-access=ALL-UNNAMED` to the konanc JVM on the subprocess (`-J<flag>` forwarding for both `konanc` and `cinterop`) and daemon (direct JVM args) paths, gated to JDK 23+ on the subprocess side (#374).
- A `kolt deps install` / `kolt deps update` now invalidates the test-build cache. `TestBuildState` (both JVM and Native) was missing a `lockfile_mtime` field, so a dependency change with no source edit could leave a stale test artifact in place — the next `kolt test` would short-circuit on the Native target or skip recompile on others. The schema now mirrors the main `BuildState` lockfile signal. The schema field is `lockfile_mtime` next to `test_artifact_mtime` (renamed from `test_kexe_mtime` so the same cache file works for both targets). Pre-v1, no migration: existing `build/.kolt-test-state.json` files lack the new field and decode to null on first read after upgrade, forcing a single cold rebuild — subsequent runs short-circuit normally (#373).

**Behavior**

- `kolt test` now caches up-to-date results on the JVM target and short-circuits the entire pipeline (no kotlinc, no JUnit launcher) when nothing has changed. Previously every warm `kolt test` re-fired kotlinc + the launcher, costing ~7.5s on a hello-world project — a hello-world JVM warm `kolt test` is now ~0.02s end-to-end. Cache hit prints `<name> tests are up to date (Xs)`, mirroring `kolt build`'s up-to-date line. Native `kolt test` previously cached the rebuild but always ran the kexe; cache hit now also short-circuits the kexe run, so warm `kolt test` on Native prints only the up-to-date line and no `running tests...` / test pass count. The JVM test compile stays on the subprocess kotlinc path; routing it through the JVM compile daemon is tracked separately (#373; daemon route follow-up #376).
- `kolt add` happy-path output reorders from `added → resolving → fetch complete` to `resolving → added → fetch complete`. Same set of lines, different order — the `added` line now means "added and persisted." Watch-mode parsers and CI scripts that key on these strings should be unaffected (no order assumption was found in-tree), but flagging in case any external tooling cares (#353).
- `kolt new` / `kolt init` now scaffold a `.gitignore` covering `build/`, `workspace.json` (env-dependent absolute paths in the LSP workspace file), `.idea/`, `*.iml`, and `.DS_Store` — previously only `build/` was listed, so first commits typically dragged in IDE state and `workspace.json`. Existing `.gitignore` files are still left untouched (#362).
