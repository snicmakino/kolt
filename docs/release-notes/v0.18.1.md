**Behavior**

- `kolt test` (JVM) routes test compile through the JVM compile daemon, same as `kolt build`. With the daemon hot, an incremental test rebuild drops from ~6 s (subprocess) to ~490 ms — roughly 12× faster on the iterative-edit case. Daemon-side IC state is segmented per compile scope (`~/.kolt/daemon/ic/<v>/<projectId>/{main,test}/`), so test compile no longer wipes `build/classes/` mid-build (#376).
- `kolt test` caches up-to-date results and short-circuits the entire pipeline (no kotlinc, no JUnit launcher) when nothing has changed. Warm `kolt test` on a hello-world JVM project is now ~0.02 s end-to-end. Native warm `kolt test` also short-circuits the kexe run; cache hit prints only the up-to-date line (#373).
- Stale daemons recover automatically. After upgrading, the first build against a v0.18.0-or-older daemon prints one stderr line and falls back to subprocess; the next build spawns a fresh daemon and runs warm. No manual `kolt daemon stop` is required (#379).
- BTA shrunk classpath snapshots are shared across compile scopes via a content-keyed cache at `~/.kolt/daemon/ic/<v>/shrunk-snapshots/`. The cache also survives daemon restarts; previously the per-jar classpath snapshot dir was being silently regenerated on every daemon restart (#380).
- `kolt test` warm output suppresses the `<name> is up to date` line that would otherwise appear right before `compiling tests...` on JVM. Standalone `kolt build` / `kolt check` still print the cache-hit confirmation (#359).
- `kolt new` / `kolt init` scaffold a `.gitignore` covering `build/`, `workspace.json`, `.idea/`, `*.iml`, and `.DS_Store`. Existing `.gitignore` files are left untouched (#362).
- `kolt add` happy-path output reorders to `resolving → added → fetch complete`. The `added` line now means "added and persisted" (#353).

**Bugfix**

- `kolt add` no longer mutates `kolt.toml` when the fetch fails. A typo or unfetchable coordinate (404, dead repo) used to leave the project locked into a bogus entry until the user ran `kolt remove` (#353).
- `kolt add <ga>` (no version) no longer auto-picks pre-releases. The pre-fix code trusted Maven Central's `<release>` tag, which surfaces `-rc`, `-alpha`, `-beta`, `-M\d+`, `-eap`, `-dev`, `-preview` qualifiers. If every published version is pre-release, kolt falls back with a `warning: no stable release found ...` to stderr instead of blocking (#354).
- Failed downloads now name the URL and HTTP status (or network-error message) for every repository tried, instead of collapsing to a bare `failed to download <ga>`. Empty `[repositories]` surfaces a config-fix hint rather than a network-style message (#355).
- `kolt remove` of the last entry in a section drops the now-empty section header. A bare `[dependencies]` no longer dangles in the diff (#360).
- Unknown subcommand exits `127` (`Command not found`) instead of `1` (`Build error`), matching the README exit-code table. CI scripts that grep for `$? == 1` to gate on a typo'd `kolt foo` should switch to `127` (#361).
- `kolt fmt` no longer leaks the 4-line `sun.misc.Unsafe::objectFieldOffset` deprecation warning ktfmt's bundled intellij-util triggers on JDK 23+ (#358).
- `kolt build` / `kolt test` on a Native target no longer leak `sun.misc.Unsafe` and `java.lang.System::load` warning blocks from konanc on JDK 23+. A project with two `[[cinterop]]` blocks emitted 32 lines of noise per fresh `kolt build` before; now silent (#374).
- `kolt fetch` materialises `[classpaths.X]` bundle jars when the lockfile-trusted reuse path runs against a clean cache. Previously a clean `~/.kolt/cache` plus an existing lockfile produced bundle classpaths pointing at non-existent jars, so the next `kolt build` failed with `Unresolved reference` (#387).
- `[classpaths.X]` bundles whose declared coordinate redirects via Gradle Module Metadata (e.g. `kotlinx-coroutines-core` → `-core-jvm`) now resolve the cached jar from the redirect target. Existing lockfiles with redirect-affected bundle entries need a one-time `rm kolt.lock && kolt fetch` to repopulate the new `redirect_target` lockfile field (#386).
- `kolt deps install` / `kolt deps update` now invalidates the test-build cache. `TestBuildState` was missing a `lockfile_mtime` field, so a dependency change with no source edit could leave a stale test artifact in place. Existing `build/.kolt-test-state.json` files lack the new field and force a single cold rebuild on first read after upgrade; subsequent runs short-circuit normally (#373).
- `kolt test` invokes JUnit Console Launcher's `execute` command explicitly. The bare-form invocation triggered a `WARNING: Delegated to the 'execute' command` banner on every test run with JUnit Platform Console Launcher 1.11+ (#357).
- `kolt workspace.json` sdk fallback honors the kolt-managed bootstrap JDK (ADR 0017) when PATH has no `java` and `[build] jdk` is unset, instead of warning that no JDK was found. The old warning fired on every `kolt add` / `fetch` / `build` even when bootstrap was successfully driving the build (#356).
