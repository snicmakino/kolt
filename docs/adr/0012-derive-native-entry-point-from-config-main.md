# ADR 0012: Derive the Kotlin/Native entry point from `config.main`

## Status

Accepted (2026-04-13)

## Context

For `target = "jvm"`, `kolt.toml` requires a `main` field that is the
fully-qualified name of a JVM class â€” by convention, the synthetic
facade class kotlinc generates for top-level functions in a file:

```toml
main = "com.example.MainKt"
```

The JVM build path uses this value directly as the argument to
`java -cp ... <main>`. The class `MainKt` is generated by kotlinc from
`Main.kt` and contains the top-level `fun main()` as a static method.

For `target = "native"`, konanc needs something different: the
fully-qualified name of the **function** itself, not a containing
class. There is no JVM facade class in a native executable. The same
project that runs as `java -cp ... com.example.MainKt` on the JVM must
be invoked as `konanc -p program -e com.example.main` for native.

The user's config has only one slot for "where my main lives", and the
question is whether `target = "native"` should:

1. Reuse `config.main` and derive the native entry point from it, or
2. Add a separate field (e.g. `native_entry_point` or `entry_point`) so
   the user can spell it out explicitly.

A spike during Phase A confirmed the relationship: for a project with
`fun main()` in `Main.kt` declaring `package com.example`, the JVM
class is `com.example.MainKt` and the native entry point function is
`com.example.main`. The package portion is identical; only the trailing
segment differs (`MainKt` â†’ `main`).

## Decision

Reuse `config.main` for both targets. For native builds, derive the
entry point FQN by stripping the class-name segment and appending
`.main`:

```kotlin
internal fun nativeEntryPoint(config: KoltConfig): String {
    val lastDot = config.main.lastIndexOf('.')
    val pkg = if (lastDot >= 0) config.main.substring(0, lastDot) else ""
    return if (pkg.isEmpty()) "main" else "$pkg.main"
}
```

This is passed to konanc via `-e <derived>`:

```
konanc src -p program -e com.example.main -o build/<name> -l ...
```

The derivation has two assumptions:

1. The entry function is named `main` (the standard Kotlin convention).
2. The function lives at the top level of the same package as the JVM
   facade class.

When `config.main`'s last segment does not end in `Kt` (the conventional
kotlinc suffix for file-level facades), kolt emits a warning before the
build, on the assumption that the user is pointing at a non-top-level
entry that the heuristic will not match correctly:

```kotlin
internal fun needsNativeEntryPointWarning(config: KoltConfig): Boolean {
    val lastSegment = config.main.substringAfterLast('.')
    return lastSegment.isNotEmpty() && !lastSegment.endsWith("Kt")
}
```

The warning fires only on native builds (in `doNativeBuild`); the JVM
path is unaffected because JVM uses `config.main` verbatim.

For `kolt test` on native, the entry point is **not** derived. konanc's
`-generate-test-runner` flag generates a synthesised entry point during
the test-runner lowering pass, so `nativeTestBuildCommand` omits `-e`
entirely (see PR #58 / Phase D).

## Consequences

### Positive

- **Single source of truth in `kolt.toml`.** Users do not have to keep
  two related fields in sync. A project that switches from
  `target = "jvm"` to `target = "native"` (or maintains both via local
  experiments) does not change `main`.
- **No new config schema.** Existing `kolt.toml` files work for native
  builds with no migration. The config structure stays minimal.
- **Convention matches the standard Kotlin layout.** For the >95% of
  projects that put `fun main()` at the top level of `Main.kt`, the
  derivation is correct without any user action.
- **Diagnosable failure mode.** When the heuristic does not match user
  intent, `needsNativeEntryPointWarning` flags it before the build, and
  konanc's own "entry point not found" error fires after that â€” both
  visible to the user.

### Negative

- **Heuristic fragility.** Users who name their entry function
  something other than `main` (rare but legal in Kotlin via
  `@kotlin.jvm.JvmName` or just `fun start()`) will silently get the
  wrong derived FQN, hit konanc's "entry point not found" error, and
  have to read the warning to figure out what happened.
- **Implicit coupling between JVM convention and native derivation.**
  `config.main` is documented as a JVM class name. Future changes to
  what we accept there (e.g. allowing a function FQN directly for JVM
  via `-Xmain-function`) would have to revisit the derivation rule.
- **No way to override.** A user with a non-standard layout cannot tell
  kolt "ignore the heuristic, use this exact FQN for native". They
  would have to rename their function to `main` or restructure their
  package.
- **`*Kt` warning is a name-shape heuristic, not a real check.** A user
  with a perfectly valid `com.example.AppKt` referencing
  `fun runApp(): Unit` (not `main`) gets no warning, builds, and only
  sees the konanc error after compilation starts.

### Neutral

- **Test entry point is unaffected.** `kolt test` on native does not
  use `nativeEntryPoint` at all â€” konanc's `-generate-test-runner`
  generates the entry point during lowering. The derivation rule only
  applies to `kolt build` / `kolt run` for the production binary.

## Alternatives Considered

1. **Add a separate `entry_point` field to `kolt.toml`** â€” rejected
   because (a) it doubles the user's surface area for the >95% case
   that follows the convention, (b) it creates a redundant field for
   the JVM target where it would just duplicate `main`, and (c) it
   moves the burden of "knowing the package" onto the user, which is
   information they already provided in `main`.
2. **Require `config.main` to be the function FQN for both targets,
   change the JVM path to `kotlinc -Xmain-function=...`** â€” rejected
   because it breaks every existing `kolt.toml` and removes the natural
   correspondence with how `java` is invoked on the command line.
3. **Auto-detect the entry function by scanning sources for
   `fun main()`** â€” rejected because it adds source-level analysis to
   what is otherwise a config-driven build tool, fails on multi-file
   projects with multiple `main` functions, and conflicts with cases
   where the user wants a specific entry function chosen.
4. **Always require explicit configuration on native, leave JVM as
   today** â€” rejected as the worst of both worlds: forces extra config
   on the more cumbersome target while the easier target is implicit.
5. **Use konanc's default entry point search instead of `-e`** â€”
   rejected because konanc defaults to the root package, and projects
   that put `fun main()` in a non-root package (the standard layout)
   would fail with "Could not find 'main' in '<root>' package". This
   was the actual error encountered during Phase A's first end-to-end
   run, which is what motivated adding `-e` in the first place.
