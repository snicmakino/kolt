# kolt Dogfooding Log

Append-only log of usability findings, surprises, and friction surfaced
while using kolt to develop kolt itself. One line per entry — link to a
GitHub issue if filed, otherwise just record the observation. The aim
is to catch "noticed and forgot" before it rots.

## Format

`YYYY-MM-DD <observation> → <outcome>`

Outcomes: `#N` (issue filed), `fixed in #N` / `landed in #N`, `wontfix`,
`watching` (observed; not yet acted on).

Issues surfaced via this channel get the `dogfood` label so they're
distinguishable from external-user reports.

## 2026-04

- 2026-04-30 `kolt test` GTEST output is ~2k lines for the full kolt suite, drowning CI logs → fixed in #301 (`--ktest_logger=SILENT` + `::group::`).
- 2026-04-30 `doRun` / `doTest` / `doBuild` hold the project lock for the full pipeline including child-process lifetime; tests calling these in-process under self-host `kolt test` deadlock at the 30s timeout → #303.
- 2026-04-30 Bootstrap kolt placed at `$GITHUB_WORKSPACE/bootstrap-kolt` with no `libexec/` sibling makes `DaemonJarResolver` miss the daemon jars; bootstrap-driven CI steps fall back to subprocess compile, costing ~30–60s per run → #302.
- 2026-04-30 `KOLT_BOOTSTRAP_TAG` self-bootstrap means a broken pinned tag blocks the next release — pin must always trail HEAD by ≥ 1 published release → policy noted in workflow headers, no issue.
- 2026-04-30 JVM daemon's 3s spawn-to-connect retry budget is too short for cold JVM startup on CI; first build after a fresh boot falls back to subprocess even though the daemon spawn itself succeeds. Surfaced via #302's canary expecting "no fallback at all" — instead the canary had to be relaxed to "daemon code path was reached" → #310.
- 2026-04-30 `kolt.lock` v3 → v4 migration on the jvm-sys-props branch (#318): `kolt deps install` printed the warning and overwrote cleanly in both daemon subprojects. Root project has no `kolt.lock` (native target, deps via cinterop / native resolver), so only two files moved. Diff is just the `"version"` line — `classpath_bundles` defaults to empty and is omitted by the serializer. Builds and tests stayed green post-migration; the Option B "deps install only" policy from design.md held.

## 2026-05

- 2026-05-01 `kolt test -- --select-class=<FQCN>` fails: JUnit Platform Console Launcher 1.11.4 rejects `--scan-class-path` together with explicit selectors (`Scanning the classpath and using explicit selectors at the same time is not supported`). `testRunCommand` always inserts `--scan-class-path`, so trailing-arg passthrough cannot be used to narrow execution to a single class or package. Surfaced during the daemon-test-via-kolt (#315) Pre-flight Gate → #323 (fixed).
- 2026-05-03 `kolt add <group:artifact>` (no version) resolves to the latest *published* version including pre-releases — `kolt add org.jetbrains.kotlinx:kotlinx-coroutines-core` selected `1.11.0-rc02`. `kolt-usage` describes this path as "latest stable", so either the resolver should filter `-rc` / `-alpha` / `-beta` / `-M*` qualifiers (and probably any non-empty Maven qualifier) or the docs need to drop the "stable" promise → fixed in #364.
- 2026-05-03 `kolt add com.example:does-not-exist:1.0.0` writes to `kolt.toml` *before* the resolver checks fetchability. Once written, every subsequent `kolt build` / `test` / `fetch` exits 3 on the same download error until the user runs `kolt remove`. Cargo validates first, then writes — same order would prevent a one-typo lockout → fixed in #363.
- 2026-05-03 Resolver download-failure message is `error: failed to download <group:artifact>` with no repository URL, HTTP status, or attempted-URLs list. Distinguishing typo vs. private-repo auth vs. transient 5xx requires re-running under a debugger or `strace` → fixed in #365.
- 2026-05-03 JDK warning `workspace.json sdk homePath left unset — could not locate 'java' on PATH (install a JDK or set [build] jdk)` fires on every `add` / `fetch` / `build` even when kolt is successfully driving its auto-fetched bootstrap JDK. The message contradicts the observed success and primes new users to install a JDK they don't actually need → fixed in #369.
- 2026-05-03 `kolt test` leaks `WARNING: Delegated to the 'execute' command. This behaviour has been deprecated and will be removed in a future release. Please use the 'execute' command directly.` from JUnit Platform Console Launcher on every run. Switching the launcher invocation from the bare form to `execute` silences it without functional change → fixed in #371.
- 2026-05-03 `kolt fmt` / `kolt fmt --check` leaks four lines of `sun.misc.Unsafe` deprecation warnings from ktfmt 0.54 under JDK 25 on every invocation. Bump ktfmt, or filter the launcher stderr → fixed in #370.
- 2026-05-03 `compiling tests...` is printed immediately after `<name> is up to date (0.0s)` during `kolt test`, which reads as contradictory — the "up to date" claim is about the main artifact, but the test artifact still needs compilation. Either suppress the main-artifact line in the test pipeline, or rephrase so the scope is explicit → fixed in #372.
- 2026-05-03 Single-line edit warm rebuild on a freshly-scaffolded hello-world JVM project measured ~1.8–2.0s across 3 back-to-back runs (daemon already hot, deps cached, no-change run is ~25ms). ADR 0016 advertises "~0.3 s warm rebuild" but a vanilla `kolt new && kolt build && edit && kolt build` does not reproduce it. Either spell out the ADR's measurement conditions (kotlinc daemon already warm in same shell, source size, hardware) or take a fresh perf pass → watching.
- 2026-05-03 Warm `kolt test` on JVM target measures ~7.5s vs ~24ms on Native (~300x gap) on a hello-world probe. Earlier diagnosis ("JUnit Console Launcher JVM cold start dominates") was wrong — `kolt test --no-daemon` produces the same ~7.5s, proving the JVM compile daemon isn't being used at all. Real cause: `doTestInner`'s JVM branch (`BuildCommands.kt:1056`) spawns kotlinc as a fresh subprocess every time, and there's no JVM analog of Native's `.kolt-test-state.json` up-to-date short-circuit. Native test pipeline already has both (daemon route + cache check) → #373.
- 2026-05-03 `kolt fmt` warm run (ktfmt jar already cached at `~/.kolt/tools/`) measures ~2.0–2.5s on a two-file project — JVM cold start + ktfmt classloading. The original cold reading of ~9.2s was misleading: it bundled the first-run `downloading ktfmt...` step. 2s is borderline-tolerable for pre-commit; not actionable on its own → wontfix-for-now (reconsider only if a bigger perf pass touches the formatter path).
- 2026-05-03 `kolt remove <group:artifact>` leaves an empty `[dependencies]` table header in `kolt.toml`. Cosmetic only, but makes the file read as "still has dependencies" at a glance and looks unfinished in PR diffs → fixed in #366.
- 2026-05-03 Unknown subcommand (`kolt foobar`) exits 1; the README exit-code table reserves 127 for "Command not found". Either the table is shell-only (kolt binary absent) and should say so, or unknown-subcommand should map to 127. Doc/impl alignment either way → fixed in #367.
- 2026-05-03 Generated `.gitignore` contains only `build/`; first-time IntelliJ users will end up checking in `.idea/` and `*.iml`. Adding the standard JetBrains + OS-junk lines (`.idea/`, `*.iml`, `.DS_Store`) to the scaffold is a one-line UX win → fixed in #368.
- 2026-05-03 `kolt build` and `kolt test` on Native targets (e.g. `linuxX64`) leak ~14 lines of JVM warnings from `kotlin-native-compiler-embeddable.jar` per invocation under JDK 23+: a `java.lang.System::load` restricted-method warning and a `sun.misc.Unsafe::objectFieldOffset` deprecation warning, each repeated twice (klib stage + kexe link stage). Same flavor as #358's ktfmt fix, but the konanc launcher (subprocess + daemon) doesn't pass `--sun-misc-unsafe-memory-access=allow` or `--enable-native-access=ALL-UNNAMED` → #374.
- 2026-05-04 #373 split: cache work shipped as #373's PR, daemon-route work tracked as #376. The original issue conflated two independent fixes; review surfaced that routing JVM test compile through the daemon corrupts `build/classes/`. Root cause: `BtaIncrementalCompiler.moduleNameFor` ignores `request.moduleName` and returns `kolt-${projectId}`, so main and test compile share both the module name and the daemon-side `btaWorkingDir` (per `IcStateLayout.workingDirFor`, derived from projectRoot only). BTA reads "same module, output relocated" and wipes the prior outputDir. Fix requires ADR 0019 §5 revision (the "1 project = 1 module = 1 outputDir" assumption) → tracked as #376.
- 2026-05-04 #373 fix verified on `/tmp/kolt-373-probe` (hello-world JVM): warm `kolt test` with no source change is `0.019s` end-to-end vs pre-fix `~7.5s` (~400× faster, sub-second budget hit with ~50× margin). `build/.kolt-test-state.json` materialises after the cold run; warm run prints `<name> tests are up to date (0.0s)` and skips kotlinc + JUnit launcher entirely. Test compile stays on the subprocess kotlinc path (daemon route deferred to #376). Native `kolt test` aligned to the same full short-circuit on cache hit, removing the long-standing asymmetry with `kolt build` → fixed in #373's PR.
- 2026-05-04 #376 root cause refined and fixed: BTA's `inputsCache` lives under `workingDir/inputs/` keyed by source-file path. Sharing one workingDir across main and test compiles makes BTA see the *other* compile's sources as "removed" and call `removeOutputForSourceFiles` against the previously-tracked source→class mapping, wiping `build/classes/Main.class` after a test compile (META-INF/<module>.kotlin_module is module-keyed and survives — matches the symptom in the issue). The original "moduleName collision" framing covered only one of the two roots; sharing workingDir alone is sufficient to corrupt outputs. Fix landed: `IcStateLayout.workingDirFor` appends a `<scope>` segment (`main` / `test`) so each scope owns its BTA inputsCache. (C) "honor wire `moduleName`" alone was rejected — workingDir is still shared, so the inputsCache cross-contamination remains. (D) "output-aware project identity" was not evaluated — (B) decisively resolves the symptom and (D) would only become relevant for a multi-output shape beyond `main` / `test` (KMP target matrix, future Benchmark / IntegrationTest scopes). JVM `kolt test` now routes the test compile through the daemon end-to-end. Verified on the kolt-jvm-compiler-daemon self-host smoke: 140 tests pass, `build/classes/` keeps its 92 classes, daemon `~/.kolt/daemon/ic/<v>/<projectId>/{main,test}/` populates as expected → fixed in #376's PR.
- 2026-05-04 #376 perf measurement: warm `kolt test` with daemon hot + IC warm + test source touched (incremental rebuild, the case the issue framed as "warm test rebuilds, cache miss with daemon hot") is **490ms BTA wall** vs subprocess `--no-daemon` test compile via fresh kotlinc ~6s — ~12× faster, satisfying the issue's "~1s" target with margin. Cold-path tradeoff observed: the *first* daemon-routed test compile (daemon hot, IC empty) is **11.9s BTA wall** vs subprocess ~6-8s — i.e., the daemon path is 1-5s slower than subprocess on cold IC because BTA's classpath-snapshot computation and IC state persistence overhead dominate when there is no prior state to diff against. This matches ADR 0019 §1's incremental-compile profile: warm rebuild wins, cold path is roughly equivalent or slightly worse. The win is asymmetric — accepted as the right tradeoff for the iterative-edit case the daemon is designed for.
- 2026-05-04 #376 implementation friction: routing the test compile through the daemon initially broke `IcReaper` because the original code wrote `LOCK` and `project.path` breadcrumb under `<workingDir>` — with the new `<projectIdHash>/<scope>/` layout, the reaper looking for `<projectIdHash>/LOCK` saw nothing and would have wiped a live projectIdDir on every reaper pass. Fix: relocate `LOCK` + breadcrumb to `<projectIdHash>` (the projectStateDir, parent of all scope dirs) so each project owns one of each; BTA's per-scope state stays under `<scope>/`. The reaper code did not change at all — responsibility stayed correctly partitioned between "is this project alive?" (projectIdDir-level) and "what state does this scope hold?" (scope-level). `heldLocks` map re-keyed by `projectStateDir` so main and test compiles within one daemon process share one cached `FileLock` instead of fighting an `OverlappingFileLockException`. `friendPaths` chose to land as a structured wire field rather than ride `extraArgs` — `DaemonServer.toIcRequest` strips unknown extraArgs by design (ADR 0019 §9 + #65), so an ad-hoc `extraArgs += "-Xfriend-paths=..."` would silently disappear into the daemon path while still working on subprocess fallback.
- 2026-05-04 Pre-existing bug surfaced while widening #373's cache key: `TestBuildState` (both JVM and Native) was missing `lockfileMtime`, so a `kolt deps` update with no source change would not invalidate the test artifact. Main `BuildState` already tracks lockfile mtime; the test cache schema was simply missing the field. Folded into #373's PR with an IT (touch `kolt.lock` ⇒ warm `kolt test` rebuilds) → fixed in #373's PR.
- 2026-05-04 `doBuildInner` and `doTestInner` walk the main source tree's mtimes independently — on warm `kolt test` with no change, the main mtime walk happens twice in sequence. Cheap on hello-world (microseconds) but scales O(N) in main source count and bloats the up-to-date critical path on larger projects. Optimisation: thread `BuildResult.mainSourcesNewestMtime` (or a "main was up to date" flag) from `doBuildInner` into `doTestInner` so the cache check can reuse it → watching, file when it shows up on a real project.
- 2026-05-06 #380 perf measurement: cross-scope shrunk-classpath-snapshot cache landed end-to-end (`ShrunkClasspathSnapshotCache` + `BtaIncrementalCompiler` hooks + `Main.kt` wiring + `IcReaper` skip-set). Spike T1 at `spike/bta-shrunk-portability/REPORT.md` is the canonical cache-effect measurement: manual placement of main scope's shrunk snapshot at test scope's per-scope path produced **78,386 ms (cache) vs 85,905 ms (no cache) median, 7,519 ms / 8.7% reduction**, clearing the OQ-1 GO threshold (>=5%). Spike Q3 also closed OQ-3 by confirming BTA writes the shrunk file in-place (inode preserved across compiles), so `Files.copy` is the permanent placement strategy; hardlink would corrupt the cache file via the shared inode.
- 2026-05-06 #380 follow-up perf measurement: end-to-end `kolt test` wall on the `kolt-jvm-compiler-daemon` self-host fixture (167 tests) does NOT surface the 8.7% cache-effect figure. arm-off median ~105 s vs arm-on median 108 s — both within the run-to-run noise envelope (~5 s). Two confounds: (a) JUnit suite execution dominates (~80 s constant) so the ~7 s BTA-compile improvement is inside the noise, (b) wiping `<v>/shrunk-snapshots/` between iterations does not actually disable the cache because `kolt test` re-runs `kolt build` internally when `.kolt-state.json` is missing, immediately re-populating the cache *inside* the same kolt-test invocation. Daemon-log inspection confirms both arms see the same `lookup=miss -> store=success -> lookup=prefix_hit` sequence. The bench at `spike/bench-scaling/cold-test-after-main.sh` is therefore not a useful regression gate as written — the only signal it provides is "no end-to-end regression observed". For Req 3.1 (warm rebuild <=540 ms BTA wall) and Req 3.2 (no-op test <=50 ms BTA wall), end-to-end kolt-test wall is the wrong measurement scope on this fixture (the targets are BTA-internal numbers from #376 dogfood, not user-observed wall) — the bench measured 89 s warm-rebuild and 88 s no-op, dominated by JUnit. Recommendation tracked: extract BTA-wall from daemon log or swap fixture for a smaller scaffold if this bench needs to become a real regression gate.
- 2026-05-08 `kolt build` / `kolt add` / `kolt fetch` print `resolving dependencies...` then go silent through the actual POM and jar downloads. On a project with many deps the silence can run 10 s+ with no progress signal — user cannot tell if kolt is fetching, stuck on a slow mirror, or hung on DNS. Per-artifact progress during the fetch loop (URL or `[N/M]` counter, optionally byte counts) would close the gap. Distinct from #2 — that issue is error-rendering scope; this is long-operation progress UX → #404.
- 2026-05-08 `install.sh` runs essentially silent through JDK + kotlinc + kolt-binary downloads despite each being multi-MB and several seconds — same "long operation, zero feedback" UX gap as the in-Kotlin resolver but in bash. Add an `echo` per download step (`fetching <name>...`) plus a final byte-count line per fetch. Separate workstream from the in-Kotlin progress fix because `install.sh` is shell-only → #405.
- 2026-05-09 `kolt new` native-target prompt orders `NATIVE_TARGETS` alphabetically (`Prompt.kt:58` `PROMPT_TARGETS = listOf(jvm) + NATIVE_TARGETS.toList().sorted()`), so the first native option a user sees is `linuxArm64` even though `linuxX64` is the typical KMP / server default. A user-mix or tier-aware order (e.g. macosArm64 → linuxX64 → linuxArm64 → macosX64 → mingwX64) would land "press the first native number" on a more-likely-correct target. Whether kolt's build pipeline should support iOS / Android native is a separate, much bigger question; the prompt-order tweak is independent and small → #411 (broader roadmap thread covering target-model direction); prompt-order tweak landed in #412 (4-preset top-level + share-ordered native sub-prompt + `(deprecated)` suffix on `macosX64`, native target set unchanged at 5).
